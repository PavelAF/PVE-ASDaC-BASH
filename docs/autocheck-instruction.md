# Инструкция: Автопроверка стендов

Инструмент для автоматизированного сбора информации с ВМ развёрнутых стендов. Выполняет предопределённые команды на ВМ и выводит сырой результат. Оценка правильности — на усмотрение преподавателя.

## Требования

- **Proxmox VE** с развёрнутыми стендами (пулы с ВМ)
- **QEMU Guest Agent** — установлен и запущен внутри ВМ (для `exec_agent`)
- **socat** — на хосте PVE (для `exec_serial`, опционально)

## Запуск

### Через меню

1. Запустите скрипт: `./PVE-ASDaC-BASH.sh`
2. Выберите группу стендов
3. В меню управления стендами выберите пункт **11. Автопроверка стенда**
4. Выберите файл конфигурации из списка (или введите путь)
5. Выберите стенд(ы) для проверки
6. При нескольких стендах — укажите, сколько проверять одновременно (по умолчанию 1)
7. При необходимости: ответьте y/1 на вопрос «Сохранить результаты в файл?», затем укажите путь (по умолчанию — текущая директория)
8. Дождитесь завершения — результаты выводятся по каждой проверке

### Через командную строку

```bash
./PVE-ASDaC-BASH.sh --autocheck autocheck/09.02.06-2026_module1.conf
```

Или с указанием URL:

```bash
./PVE-ASDaC-BASH.sh --autocheck http://172.16.99.99/mirror/PVE-ASDaC-BASH/check.conf
```

### Режим отладки (verbose)

```bash
./PVE-ASDaC-BASH.sh --autocheck check.conf -v
```

Выводит подробную информацию о пинге агента, попытках выполнения и таймаутах.

## Файлы конфигурации

### Расположение

```
./PVE-ASDaC-BASH.sh
./autocheck/
    ├── example.conf           # Пример формата
    ├── 09.02.06-2026_module1.conf
    └── custom.conf
```

Конфиг можно указать по локальному пути или URL (HTTP/HTTPS).

### Формат конфига

Файл `.conf` — bash-скрипт с переменными:

```bash
# Название автопроверки (отображается в заголовке)
autocheck_name='ДЭ 09.02.06-1-2026 Модуль 1'

# ВМ и способ подключения
#   exec_agent  — QEMU Guest Agent (qemu-guest-agent в ВМ)
#   exec_serial — serial console (socat + serial0)
autocheck_vms='
    ISP    = exec_agent
    HQ-RTR = exec_agent
    HQ-SRV = exec_agent
    HQ-CLI = exec_agent
'

# Проверки
check_1_name='Имя хоста'
check_1_vms='ISP HQ-RTR HQ-SRV HQ-CLI'
check_1_cmd_exec_agent='hostname -f 2>/dev/null; hostname'

check_2_name='IP-адреса'
check_2_vms='ISP HQ-RTR HQ-SRV HQ-CLI'
check_2_cmd_exec_agent='ip -br a'
```

### Переменные проверки

| Переменная | Описание |
|------------|----------|
| `check_N_name` | Название проверки |
| `check_N_vms` | Список ВМ через пробел (ISP, HQ-RTR, …) |
| `check_N_cmd_exec_agent` | Команда для ВМ с exec_agent |
| `check_N_cmd_exec_serial` | Команда для ВМ с exec_serial |

### Serial console (опционально)

Для ВМ с `exec_serial`:

```bash
exec_serial_prompt='[A-Za-z0-9._-]+[#>]'
exec_serial_timeout='3'
exec_serial_login='admin'
exec_serial_password='P@ssw0rd'

# Переопределение для конкретной ВМ (дефис → подчёркивание)
# HQ_RTR_serial_login='root'
# HQ_RTR_serial_password='toor'
```

## Маппинг ВМ

Имена ВМ из конфига (ISP, HQ-RTR, …) сопоставляются с полем `name` ВМ в пуле PVE. Имена должны совпадать.

## Как это работает

1. **Пинг агента** — перед опросом каждой ВМ проверяется доступность Guest Agent
2. **Запись скрипта** — все команды для ВМ собираются в один скрипт и записываются на ВМ через `agent/file-write`
3. **Выполнение** — скрипт запускается одной короткой командой (снижает нагрузку на агент)
4. **Пауза между ВМ** — 3 секунды, чтобы не перегружать гипервизор

Для слабых ВМ (1 ядро) этот подход стабильнее, чем множество отдельных вызовов агента.

## Ошибки и решения

| Сообщение | Причина | Решение |
|----------|---------|---------|
| Guest Agent недоступен | Агент не отвечает на ping | Установить qemu-guest-agent в ВМ, перезагрузить ВМ |
| got timeout | ВМ перегружена, агент не успевает | Увеличить паузу, перезагрузить ВМ перед проверкой |
| ВМ не найдена в стенде | Имя ВМ в конфиге не совпадает с name в пуле | Проверить имена в конфиге и PVE |
| ВМ не запущена | ВМ выключена | Запустить ВМ |
